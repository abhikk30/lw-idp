# lw-idp P2.2 — observability in the IDP UI (logs + traces + metrics + pods)

**Status:** approved (2026-04-29)
**Predecessors:**
- `2026-04-25-lw-idp-p1.8-observability-design.md` (Loki, Tempo, Grafana, Prometheus — the data layer is already running)
- `2026-04-29-lw-idp-sample-nginx-design.md` (P2.1.2 — first dogfooded user app, gives us a real workload to point the panels at)

## 1. Goal

Make the IDP service detail **Overview tab** the canonical pane for "what's happening with this service right now" — without leaving the IDP for Grafana — by surfacing every observability signal we already have running:

- **Logs** (Loki) — last 1h, polling every 15s, with `trace_id` rendered as a link.
- **Traces** (Tempo) — most recent 10 traces touching the service.
- **Metrics** (Prometheus) — request rate, error %, p95 latency over 1h as sparklines.
- **Pods** (Kubernetes API) — replicas ready, ages, restart counts.

Every panel is linked: clicking a `trace_id` in a log line opens the trace drawer; clicking a span in the trace drawer filters the logs panel; clicking "Open in Grafana" on any panel deep-links to the same query in Grafana for ad-hoc analysis.

## 2. Non-goals

- **Live tail / streaming** — polling at 15s is the agreed cadence (sub-100rps services don't benefit from streaming).
- **Time range selector** — fixed 1h. Anything richer goes via Grafana deep-link.
- **Per-pod logs** — aggregated by namespace; per-pod scoping comes via the Grafana deep-link.
- **Alerting from the UI** — Alertmanager + Grafana already cover this; we don't re-implement.
- **Drag-select on metrics graphs** — deferred. Filtering logs to a metric spike is a future enhancement.
- **Index changes in catalog-svc/identity-svc** — no schema migrations. Everything is read-side.

## 3. Architecture (one paragraph)

The gateway grows a thin observability proxy plugin (`/api/v1/observability/*`) with four read-only endpoints (logs, traces, metrics, plus `GET /api/v1/services/<id>/pods`). Each one reads the service's catalog row to find its `targetNamespace` (the same field the Argo CD App uses), then issues a single HTTP request to Loki/Tempo/Prometheus/Kubernetes API. The web UI's Overview tab is rebuilt as a 4-panel layout that fetches these endpoints client-side. Cross-linking between panels is pure UI state — no backend coordination.

## 4. The four endpoints (read-only, no DB writes)

All endpoints live behind the existing `sessionPlugin` (auth required) and the existing rate-limit middleware. None touch the catalog/identity DBs except to look up the service's `targetNamespace` from its catalog row.

### 4.1 `GET /api/v1/observability/logs`

Query params:
- `service=<uuid>` — IDP service id (required)
- `since=1h` — duration, default 1h, max 24h
- `trace_id=<hex>` — optional filter; when set, only return log lines with this `trace_id`
- `limit=200` — default 200, max 1000

Response:
```json
{
  "lines": [
    {
      "ts": "2026-04-29T13:01:02.123Z",
      "level": "info",
      "msg": "incoming request",
      "trace_id": "1a2b3c…",
      "span_id": "9f8e…",
      "pod": "sample-nginx-7d47b5b496-jz684",
      "raw": "<full log line as Loki returned>"
    }
  ],
  "truncated": false
}
```

LogQL the gateway sends to Loki:
```
{namespace="<targetNamespace>"} | json
[ | trace_id="<trace_id>" ]
```

### 4.2 `GET /api/v1/observability/traces`

Query params:
- `service=<uuid>` (required)
- `since=1h`
- `limit=10`

Response:
```json
{
  "traces": [
    {
      "trace_id": "1a2b…",
      "root_service": "gateway-svc",
      "root_operation": "GET /api/v1/services",
      "started_at": "2026-04-29T13:01:02.000Z",
      "duration_ms": 47,
      "span_count": 6,
      "status": "ok"  // or "error"
    }
  ]
}
```

TraceQL the gateway sends to Tempo:
```
{ resource.service.name="<svc-name>" } | duration > 0
```
(`<svc-name>` derived from the catalog row's `slug`; matches the OpenTelemetry resource attr the services already emit via `service-kit`.)

### 4.3 `GET /api/v1/observability/traces/<trace_id>`

Returns the full spans tree for the trace drawer:
```json
{
  "trace_id": "1a2b…",
  "spans": [
    {
      "span_id": "9f8e…",
      "parent_id": null,
      "service": "gateway-svc",
      "name": "GET /api/v1/services",
      "started_at": "…",
      "duration_ms": 47,
      "status": "ok",
      "attributes": { "http.method": "GET", "http.status_code": 200 }
    }
  ]
}
```

### 4.4 `GET /api/v1/observability/metrics`

Query params:
- `service=<uuid>` (required)
- `panel=req_rate|error_rate|p95_latency` (required)
- `since=1h`
- `step=15s`

Response:
```json
{
  "panel": "req_rate",
  "unit": "req/s",
  "points": [{ "ts": "…", "value": 23.4 }, …]
}
```

PromQL per panel (each one filtered to the service's namespace; assumes services emit standard `http_server_duration_seconds_*` histograms via `service-kit` — verified live):
- **req_rate**: `sum(rate(http_server_duration_seconds_count{namespace="<ns>"}[1m]))`
- **error_rate**: `sum(rate(http_server_duration_seconds_count{namespace="<ns>",http_status_code=~"5.."}[1m])) / sum(rate(http_server_duration_seconds_count{namespace="<ns>"}[1m]))`
- **p95_latency**: `histogram_quantile(0.95, sum(rate(http_server_duration_seconds_bucket{namespace="<ns>"}[1m])) by (le))`

### 4.5 `GET /api/v1/services/<id>/pods`

Lives under the existing `services` route family for cohesion. Calls Kubernetes API server-side using the gateway's existing in-cluster ServiceAccount (which already has read on services + pods via P2.0).

Response:
```json
{
  "pods": [
    {
      "name": "sample-nginx-7d47b5b496-jz684",
      "phase": "Running",
      "ready": true,
      "started_at": "2026-04-29T12:43:11Z",
      "restart_count": 0,
      "node": "lw-idp-dev-worker"
    }
  ]
}
```

## 5. Cross-linking (the "linked panels" decision)

**`trace_id` rendered as a link in the logs panel.** When a log line's parsed JSON contains `trace_id`, the UI renders it as a small monospace badge. Clicking it sets the React Query cache key `selectedTraceId` to that value. Two effects:
1. The **traces panel** highlights the matching trace (or fetches it if not in the recent-10 list).
2. The **trace drawer** opens with that trace's spans.

**Click a span → "Show logs for this trace".** The trace drawer shows a button per-span that filters the logs panel to that `trace_id` (sets the URL search param `trace_id=<hex>`).

**Open in Grafana button on every panel.** Each panel computes a Grafana deep-link URL (logs use Loki Explore, traces use Tempo Explore, metrics use Prometheus Explore). The button opens it in a new tab. Logs deep-link includes the LogQL query; metrics include the PromQL.

## 6. Trace_id audit (Phase B)

For cross-linking to work, every lw-idp service that emits structured logs must also emit `trace_id` and `span_id` as JSON fields. The current services use `@lw-idp/service-kit` for their logging setup — service-kit ships a Pino logger that already pulls active span context (verified during P1.8). So in theory all 6 services are covered.

In practice, **log a single representative line per service** during the audit task (Phase B), grep Loki, and confirm `trace_id` is non-empty. Fix any service that doesn't emit it by adding the OTel-trace-context Pino mixin in service-kit's logger factory if it's missing for that service. The fix touches 1 file in `packages/service-kit/`.

## 8. Phasing (~13 tasks, 7 phases)

|---|---|---|

| **B. Trace_id audit** | B1 | Log one line in each of the 6 lw-idp services, query Loki, confirm `trace_id` is present. Fix the Pino mixin if any service is missing it. |
| **C. Gateway proxies** | C1, C2, C3 | C1: `/api/v1/observability/logs` + `/traces` plugin. C2: `/api/v1/observability/metrics` plugin. C3: `/api/v1/services/<id>/pods` extension to existing services plugin. |

| **G. e2e + PR** | G1, G2 | G1: e2e on the kind cluster (curl every endpoint, eyeball the page on `sample-nginx`). G2: PR. |

## 9. Surface decisions

- **No new top-level page.** Everything is inside `/services/<id>` Overview tab.
- **Static panels mode is gone** — UI always assumes the cross-linking panels.
- **Sparkline library**: recharts (small, tree-shakable, idiomatic for React Server-Components-friendly client islands). Avoid victory which is heavier.
- **Polling cadence**: logs 15s, traces 30s, metrics 30s, pods 30s. Each is independent so a slow Tempo doesn't block the page.
- **Logs panel default sort**: newest-first.
- **Truncation**: logs hard-cap at 1000 lines; UI shows "showing first 200 of 1000+" with a "Open in Grafana for full" link.
- **Errors**: each panel shows its own inline error state (e.g., "Loki unreachable") without crashing the others.

## 10. Risks worth calling out

- **Service slug ↔ OTel `service.name` mismatch.** Tempo TraceQL filters on `resource.service.name`. Our service-kit defaults `service.name` to the package name (`@lw-idp/gateway-svc` → `gateway-svc`). The catalog row's `slug` matches. Verified during Phase B audit.
- **Prometheus `http_server_duration_seconds`** is the OTel HTTP histogram (verified by curl-ing Prom in the dev cluster — series exist). If the metric name diverges between services, we may need a per-service mapping. Audited in Phase B alongside the trace_id check.
- **`namespace` label on Loki streams.** We rely on Loki's standard k8s labeller adding `namespace`. Verified — every existing Loki stream has it (P1.8).
- **Cross-linking depends on `trace_id` in logs.** If a service silently drops it, its log lines have a literal `(no trace)` badge in the UI rather than a link. Audited in Phase B.
- **`http_status_code` label cardinality.** Standard 5-bucket label, low cardinality. No risk.

## 11. Done definition

1. Visit `http://portal.lw-idp.local/services/<sample-nginx-id>` Overview tab on the dev cluster.
2. Pod strip shows `sample-nginx-…` as Running.
3. Three sparklines render with real data (curl the page in another tab during the demo to populate metrics).
4. Logs panel shows live entries; at least one has a `trace_id` link.
5. Click the `trace_id` link → trace drawer opens with the spans tree.
6. Click "Show logs for this trace" in the drawer → logs panel filters to that trace.
7. Click "Open in Grafana" on the logs panel → opens Loki Explore with the same `{namespace=…}` query pre-filled.
8. Same Overview tab works on a different service (e.g., gateway-svc itself) without code changes.
