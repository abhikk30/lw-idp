# lw-idp P2.3 — security & vulnerability reporting (Trivy Operator)

**Status:** approved (2026-05-04)
**Predecessors:**
- P2.0.5 — Argo CD Application registration; provides the slug → namespace mapping the Security tab reuses
- P2.2 — observability UI; this PR follows the same per-service-tab pattern and reuses the gateway's k8s client + Argo CD App lookup

## 1. Goal

Surface **all the security signals already collectable on a Kubernetes cluster** inside the IDP — without sending operators to a separate Trivy CLI or Grafana panel. Two surfaces:

- **Top-level Security page (`/security`)** — fleet view: cluster-wide vulnerability counts, CIS compliance, RBAC findings, exposed secrets, and a top-10 vulnerable services list.
- **Per-service Security tab** — vulnerabilities, config audits, and exposed secrets scoped to one workload. Click-through from the cluster page.

This is the IDP's first **defensive** surface — every prior tab is observational. The data exists today (Trivy Operator scans automatically); we just plumb it.

## 2. Non-goals

- **Build-time scanning gate.** Failing Jenkins builds on Critical CVEs is a separate follow-up — this PR is read-only visibility, not enforcement.
- **Admission control.** Blocking deploys with critical CVEs requires Trivy Operator's `validatingwebhook` and a clear policy story — out of scope.
- **Cross-namespace fleet for non-IDP-managed workloads.** We scan everything Trivy Operator scans (cluster-wide), but the per-service tab requires an Argo CD App matching the slug — services not registered through the IDP don't show up there.
- **Vulnerability triage / muting / suppression.** Trivy Operator supports it via annotations; the IDP UI just shows current state. Triage workflow is a follow-up.
- **CVE-detail modal.** Inline rows only in v1; a modal with CVSS/NVD link is a small follow-up.
- **No new DB schema.** All security state is read directly from Trivy's CRDs (`VulnerabilityReport`, `ConfigAuditReport`, `ExposedSecretReport`, `RbacAssessmentReport`, `ClusterComplianceReport`).

## 3. Architecture (one paragraph)

Deploy `aquasecurity/trivy-operator` Helm chart to namespace `trivy-system`. The operator watches every workload across the cluster and produces one `VulnerabilityReport` + one `ConfigAuditReport` + one `ExposedSecretReport` per pod-spec, plus one `RbacAssessmentReport` per ServiceAccount and one `ClusterComplianceReport` per CIS profile. The gateway grows a `/api/v1/security/*` plugin that reads those CRDs through the existing in-cluster k8s client (P2.2 C3); the slug → namespace lookup reuses `resolveNamespace` from the observability plugin (Argo CD App's `spec.destination.namespace`). The web UI gets a top-level `/security` page driven by the cluster-scope endpoints, plus a Security tab on the service detail page driven by the service-scope endpoint. Everything is read-only, polled every 5 minutes (Trivy scans every 6h by default).

## 4. The Trivy CRDs and what we surface

| CRD | Scope | Source field | We surface… |
|---|---|---|---|
| `VulnerabilityReport` | per-workload | `report.vulnerabilities[]` | severity counts + top 20 CVEs (CVE ID, package, installed-version, fixed-version, severity, primaryLink) |
| `ConfigAuditReport` | per-workload | `report.checks[]` | misconfigs with severity (privileged, no readonly fs, no seccomp profile, etc.) |
| `ExposedSecretReport` | per-workload | `report.secrets[]` | secret rule + match-line (rare in our setup but valuable when present) |
| `RbacAssessmentReport` | per-ServiceAccount | `report.checks[]` | RBAC audit — overly broad cluster-roles, etc. **Cluster page only** (per-service is too narrow — most lw-idp services share their SA with the namespace) |
| `ClusterComplianceReport` | cluster-scoped, one per profile | `report.summary` + `report.controlChecks[]` | CIS-Kubernetes-Benchmark pass/fail per control. **Cluster page only** |

## 5. Endpoints

All behind `sessionPlugin` auth, all GET-only.

### 5.1 Cluster scope

`GET /api/v1/security/cluster`

```json
{
  "vulnerability_summary": {
    "critical": 12, "high": 47, "medium": 213, "low": 89, "unknown": 4
  },
  "config_audit_summary": {
    "critical": 0, "high": 8, "medium": 22, "low": 5
  },
  "exposed_secrets": {
    "total": 0,
    "items": []
  },
  "scan_coverage": {
    "workloads_scanned": 18,
    "workloads_total": 18,
    "last_scan_at": "2026-05-04T01:23:00Z"
  },
  "top_vulnerable_services": [
    { "slug": "gateway-svc", "critical": 3, "high": 12 },
    { "slug": "catalog-svc", "critical": 2, "high": 9 }
  ]
}
```

`GET /api/v1/security/cluster/compliance` — full CIS benchmark breakdown:

```json
{
  "profiles": [
    {
      "name": "k8s-cis",
      "summary": { "passCount": 78, "failCount": 14 },
      "controls": [
        { "id": "5.1.3", "name": "Minimize wildcard use in Roles", "status": "FAIL", "severity": "MEDIUM" }
      ]
    }
  ]
}
```

`GET /api/v1/security/cluster/rbac` — flattened across all RbacAssessmentReports:

```json
{
  "findings": [
    {
      "service_account": "gateway-svc",
      "namespace": "lw-idp",
      "check_id": "AVD-KSV-0041",
      "title": "Manage all secrets",
      "severity": "HIGH",
      "message": "ClusterRole '<x>' allows secrets:get,list,watch on '*'"
    }
  ]
}
```

### 5.2 Per-service scope

`GET /api/v1/security/services/<slug>` — bundles every per-workload report for the service's targetNamespace + its workload-name (resolved by Argo CD App lookup, same pattern as observability):

```json
{
  "service": "gateway-svc",
  "namespace": "lw-idp",
  "vulnerability_summary": { "critical": 3, "high": 12, "medium": 41, "low": 7, "unknown": 0 },
  "vulnerabilities": [
    {
      "cve_id": "CVE-2024-1234",
      "severity": "CRITICAL",
      "package": "openssl",
      "installed_version": "3.0.13",
      "fixed_version": "3.0.14",
      "primary_link": "https://nvd.nist.gov/vuln/detail/CVE-2024-1234"
    }
  ],
  "config_audits": [
    { "check_id": "AVD-KSV-0014", "title": "Root file system is not read-only", "severity": "MEDIUM", "message": "..." }
  ],
  "exposed_secrets": [],
  "last_scan_at": "2026-05-04T01:23:00Z"
}
```

Vulnerability list is capped at the top 50 (sorted Critical → High → Medium → Low). Cluster summary uses the un-capped totals.

### 5.3 Failure modes

- **Trivy Operator not installed / no CRDs** → 503 with `code: "trivy_not_installed"`. UI shows a friendly empty state with a runbook link.
- **No reports for a service yet** (operator hasn't scanned the workload) → 200 with empty arrays + `last_scan_at: null`. UI shows "Scan in progress — first results land within ~10 min after deploy."
- **Argo CD App missing for the slug** → 404 (same shape as observability).
- **k8s API unreachable** → 502.

### 6.3 Polling

Both pages use React Query with `refetchInterval: 5 * 60_000` (5 minutes). Trivy's default scan interval is 6h; this gives near-realtime feel without hammering the k8s API.

### 6.4 Severity badges (shared component)

A new `<SeverityBadge severity="CRITICAL|HIGH|MEDIUM|LOW|UNKNOWN" />` component used in both surfaces. Tailwind-styled per semantic color, no per-call overrides.

## 7. Trivy Operator deployment

New helmfile release in `infra/helmfile/observability.yaml` (kept in the same file even though Trivy isn't strictly observability — it's the umbrella for "things that scan/observe the cluster"; alternative: split into a new `infra/helmfile/security.yaml`. Decision: same file, for now).

```yaml
- name: trivy-operator
  namespace: trivy-system
  chart: aquasecurity/trivy-operator
  version: 0.31.4
  values:
    - ../trivy/values.yaml
  needs:
    - kube-prometheus-stack    # operator emits prom metrics that the existing scrape picks up
```

Values file `infra/trivy/values.yaml`:
- `targetNamespaces: ""` — scan all namespaces.
- `compliance.specs: [k8s-cis]` — start with one profile; CIS is the most actionable.
- `serviceMonitor.enabled: true` — surface Trivy's own metrics in Prometheus.
- Resource asks: `requests: { cpu: 100m, memory: 256Mi }; limits: { cpu: 500m, memory: 1Gi }` — kind-friendly.
- Scanner: default `Trivy` (the operator can also use Aqua's commercial scanner; we stay open-source).

Helm repo `aquasecurity/trivy-operator` is added to `repos.yaml` if not present.

## 8. RBAC for the gateway

New ClusterRole + ClusterRoleBinding (alongside the existing `gateway-svc-pods-reader` from P2.2):

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: gateway-svc-trivy-reader
rules:
  - apiGroups: ["aquasecurity.github.io"]
    resources:
      [
        "vulnerabilityreports",
        "configauditreports",
        "exposedsecretreports",
        "rbacassessmentreports",
        "clustercompliancereports",
      ]
    verbs: ["get", "list"]
```

Added under `charts/gateway-svc/templates/clusterrole-trivy.yaml` and applied automatically when the chart syncs.

## 10. Risks worth calling out

- **First scan delay.** Trivy Operator's first scan after install takes ~5-10 min (image pull + scan queue). The UI shows a friendly empty-state during that window — but if the user does a fresh `cluster-reset.sh dev` and immediately opens `/security`, they'll see "Scan in progress" for ~10 min. Document in the runbook.
- **CVE noise.** Open-source containers commonly carry many Medium/Low CVEs that won't be fixed for months. The UI defaults to **showing all severities** but visually de-emphasizes Low/Unknown. A future "hide Low+Unknown" toggle is a follow-up.
- **CRD-version drift.** Trivy Operator's CRD versions can change across minor upgrades (alpha API). Pin to a known-good chart version (`0.31.4`) and update with intent.
- **Scan storage.** Trivy Operator stores reports as CRDs. With ~20 workloads × 3 reports each = 60 CRDs, plus compliance/RBAC, this is fine for kind. Production clusters would want the operator's `metricsBindAddress`-only mode + external storage; out of scope here.
- **Compliance profile noise.** CIS-K8s benchmark fails on many controls in a fresh kind cluster (audit logging, etc.) that don't apply to dev. The UI shows them; a "dev profile vs prod profile" toggle is a follow-up.
- **Per-service workload mapping.** Most lw-idp services have a single Deployment matching their slug. If a service ever runs multiple workload kinds (Deployment + CronJob, say), the per-service tab merges them. We label rows with the workload kind+name in the UI to keep that explicit.

## 11. Done definition

1. `kubectl -n trivy-system get pod` shows `trivy-operator` Running.
2. `kubectl get vulnerabilityreports -A` returns ≥1 report per lw-idp service after ~10 min.
3. `GET /api/v1/security/cluster` returns 200 with non-zero `workloads_scanned`.
4. `GET /api/v1/security/services/gateway-svc` returns 200 with a non-empty `vulnerabilities` array (gateway-svc's nginx-alpine base ships with several known CVEs — guaranteed signal).
5. Sidebar nav has "Security" entry → opens `/security` page → all 5 cards render with real data.
6. Service detail "Security" tab → vulns + config audits + secrets render.
7. Click a "Top vulnerable services" row on `/security` → lands on the service's Security tab.
