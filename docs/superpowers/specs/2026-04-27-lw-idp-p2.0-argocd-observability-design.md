> **Status:** Brainstormed 2026-04-27. Ready for plan-writing.
>
> **Predecessor specs:** `docs/superpowers/specs/2026-04-20-lw-idp-foundation-design.md` (foundation; P1.x). This spec covers the first slice of Project 2 (P2 — GitOps deploys).

## 1. Goal

Install Argo CD into the dev kind cluster and integrate it with the IDP so the portal becomes the single window into "what's deployed where, at what revision, and is it healthy" — with real-time updates and the ability to trigger a Sync or a Hard Sync from inside the portal. lw-idp dogfoods Argo CD by deploying its own services through it.

P2.0 is **observability + control plane** for deploys. It does **not** introduce any new build pipeline, image registry, GitOps manifests repo, promote, rollback, or multi-cluster mechanism. Image tags get bumped manually in this repo's `charts/lw-idp/values.yaml` until P2.1 lands.

## 2. Why this slice first

- Foundation spec §10.1 already scaffolded `/services/[id]/deployments` and `/services/[id]/pipelines` with mock adapters. P2.0 swaps the mock adapter for a real one, exercising the full seam without touching the rest of the UI.
- Foundation spec §2 P3 explicitly notes "lw-idp starts dogfooding itself here" — Argo CD becomes the deploy substrate for the IDP itself, not just for tenant services.
- Promotion / multi-env / rollback all imply RBAC and policy decisions that lean into P1.9 (security hardening). Read-only observability + a couple of sync actions doesn't.
- Each later sub-plan (P2.1 build, P2.2 approve-from-IDP) layers cleanly on the surfaces P2.0 establishes (gateway proxy, NATS subjects, deploy panels in the UI).

## 3. Architecture

**Single Argo CD instance, single cluster (the dev kind one), single source repo (this monorepo).**

```
       ┌──────────────────────────────────────────────────┐
git    │                  master branch                    │
push ─►│  charts/lw-idp/values.yaml  ◄── (manual tag bump  │
       │                                  in P2.0;          │
       │                                  Jenkins+PR in P2.1)│
       └─────────────┬────────────────────────────────────┘
                     │ polled / webhooked by
                     ▼
        ┌─────────────────────────┐
        │       Argo CD           │  one ApplicationSet
        │  (argocd ns, Dex OIDC)  │  → one Application per service
        └──────┬──────────────┬───┘
               │ syncs        │ fires argocd-notifications
               ▼              ▼ webhook
       ┌────────────────┐  ┌──────────────────────────────────┐
       │  lw-idp ns     │  │ gateway /api/v1/webhooks/argocd   │
       │  (the apps)    │  └────────────┬─────────────────────┘
       └────────────────┘               │ NATS idp.deploy.*
                                        ▼
                          notification-svc → WS toast

   user click "Sync" / "Hard Sync"
       │
       ▼
   gateway /api/v1/argocd/applications/{name}/sync
       │  (forwards Dex id_token; Argo CD trusts via SSO)
       ▼
   Argo CD triggers reconcile
```

### 3.1 Three load-bearing invariants

1. **Argo CD is the system of record** for "what's deployed and is it healthy." The IDP does **not** mirror Argo CD state into its own DB. There is no `deployments` table in P2.0.
2. **All Argo CD writes are GitOps-driven.** The Sync / Hard Sync buttons trigger a *reconciliation* against Git — they do not bypass Git as the source of truth. Application definitions themselves change only via a commit to this repo (which the ApplicationSet picks up).
3. **One token model end-to-end.** Dex issues id_tokens with audience claims that span both `lw-idp-gateway` and `argocd` via `trustedPeers`. The gateway forwards the same id_token to Argo CD as a bearer; Argo CD's OIDC verifier accepts it. No token-exchange middleware to write.

### 3.2 What lives where

| Concern | Component | Notes |
|---|---|---|
| Reconciliation engine | Argo CD | helmfile-installed, in-cluster |
| Source of truth for state | Argo CD REST API | `argocd-server.argocd.svc:80` |
| Real-time state push | `argocd-notifications` controller | webhooks gateway with HMAC signature |
| State fan-out to browsers | notification-svc (existing) | wildcard subscribe `idp.>` already in place |
| User-facing API | gateway-svc proxy plugin | new module `apps/gateway-svc/src/http/argocd.ts` |
| Webhook ingestion | gateway-svc | new module `apps/gateway-svc/src/http/webhooks/argocd.ts` |
| Per-service UI | apps/web | real `DeploymentAdapter` replaces P1.7 mock |
| Cluster-wide metrics | Grafana dashboard | community ID 14584, ConfigMap-discovered |
| SSO | Dex (existing) | new staticClient `argocd` + multi-audience |

### 4.1 Argo CD install (helmfile)

**Files:**
- `infra/argocd/helmfile.yaml` (joined into top-level `infra/helmfile.yaml`)
- `infra/argocd/values.yaml`

**Helm chart:** `argo/argo-cd` (current stable; pin a known version in `helmfile.yaml`).

**Key values:**
- `dex.enabled: false` — we use the cluster's existing Dex; do not run Argo CD's bundled Dex.
- `configs.cm.url: http://argocd.lw-idp.local` — used by Argo CD for its OIDC redirect and webhook URLs.
- `configs.cm.oidc.config:` — OIDC config block pointing at our Dex:
  ```yaml
  name: lw-idp Dex
  issuer: http://dex.lw-idp.local
  clientID: argocd
  clientSecret: $oidc.dex.clientSecret  # sourced from Argo CD secret
  requestedScopes: ["openid", "profile", "email", "groups"]
  ```
- `configs.params.server.insecure: true` — dev profile only; TLS via cert-manager arrives in P1.9.
- `configs.rbac.policy.csv:` —
  ```
  p, role:lw-idp-user, applications, get, */*, allow
  p, role:lw-idp-user, applications, sync, */*, allow
  p, role:lw-idp-user, applications, action/*, */*, allow
  g, "lw-idp:authenticated", role:lw-idp-user
  ```
  All authenticated Dex users land in `lw-idp-user`. Team-scoped RBAC is deferred to P1.9 / P2.2.
- `server.ingress.enabled: true`, host `argocd.lw-idp.local`, className `nginx`, ssl-redirect false. Same pattern as Dex/portal/grafana.
- `server.ingress.hostname` is added to `scripts/cluster-bootstrap.sh`'s CoreDNS rewrite list (existing idempotent block from P1.8.2 just gets one more entry).

### 4.2 ApplicationSet (one Application per lw-idp service)

**File:** `infra/argocd/applicationset.yaml`

**Generator:** list — entries match the umbrella chart's service list.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: lw-idp-services
  namespace: argocd
spec:
  goTemplate: true
  generators:
    - list:
        elements:
          - name: gateway-svc
          - name: identity-svc
          - name: catalog-svc
          - name: cluster-svc
          - name: notification-svc
          - name: web
  template:
    metadata:
      name: '{{.name}}'
      labels:
        app.kubernetes.io/part-of: lw-idp
    spec:
      project: default
      source:
        repoURL: https://github.com/abhikk30/lw-idp.git
        targetRevision: master
        path: charts/{{.name}}
        helm:
          valueFiles: ["values.yaml"]
      destination:
        server: https://kubernetes.default.svc
        namespace: lw-idp
      syncPolicy:
        automated:
          prune: false      # destructive prune off by default; Hard Sync forces it
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
          - ApplyOutOfSyncOnly=true
```

Adding a new service = one entry under `elements:` (and a corresponding `charts/<name>/` Helm chart, mirroring the convention already used by the umbrella). P2.1 may switch the generator to discover services by globbing `charts/*/Chart.yaml` so the list is derived rather than duplicated.

### 4.3 argocd-notifications config

**File:** `infra/argocd/notifications.yaml`

Triggers (subset of stock templates):
- `on-deployed` — application reaches Synced + Healthy.
- `on-health-degraded` — health flips to Degraded.
- `on-sync-failed` — sync operation returns failed.
- `on-sync-running` — sync operation starts (gives the UI a "Progressing" toast).

Single notification service: a webhook posting to the gateway.

```yaml
service.webhook.lw-idp-gateway: |
  url: http://gateway-svc.lw-idp.svc.cluster.local:4002/api/v1/webhooks/argocd
  headers:
    - name: X-Argocd-Signature
      value: $signature
  basicAuth: ""
```

Each template renders a JSON body:
```json
{
  "app": "{{.app.metadata.name}}",
  "revision": "{{.app.status.sync.revision}}",
  "syncStatus": "{{.app.status.sync.status}}",
  "healthStatus": "{{.app.status.health.status}}",
  "operation": "{{.app.status.operationState.phase}}"
}
```

HMAC: `argocd-notifications` does not natively HMAC bodies. We add it via a `templated` header that prepends a shared secret + body hash — implemented in the template, verified in the gateway. (Alternative if too brittle: leverage Argo CD's `ConfigMap`-defined Go template `sprig` functions to compute HMAC. Plan should include a fallback to a short shared bearer token if HMAC-via-template proves fragile.)

### 4.4 Dex changes

**File:** `infra/dex/values.yaml` (existing)

Add a new staticClient and multi-audience:

```yaml
staticClients:
  - id: lw-idp-gateway
    name: lw-idp Gateway
    redirectURIs: ["http://portal.lw-idp.local/auth/callback"]
    secret: devnotreal
    trustedPeers:
      - argocd                # tokens for lw-idp-gateway also valid for argocd
  - id: argocd
    name: Argo CD
    redirectURIs: ["http://argocd.lw-idp.local/auth/callback"]
    secret: devnotreal-argocd
    trustedPeers:
      - lw-idp-gateway        # symmetric
```

`trustedPeers` causes Dex to put both audiences into the issued token's `aud` array, so a single id_token validates against either client.

The `argocd` client secret is materialized as a Kubernetes Secret in the `argocd` namespace (`argocd-secret` key `oidc.dex.clientSecret`) by `scripts/cluster-bootstrap.sh`. Same idempotent pattern as the other Dex secrets.

### 4.5 gateway-svc — Argo CD proxy plugin

**File:** `apps/gateway-svc/src/http/argocd.ts` (new)

**Routes:**

- `GET /api/v1/argocd/applications` → list (filtered to `app.kubernetes.io/part-of: lw-idp`).
- `GET /api/v1/argocd/applications/:name` → full app status object.
- `POST /api/v1/argocd/applications/:name/sync` → triggers reconcile.
  - Body: `{ prune?: boolean, force?: boolean }`. Defaults `prune=false, force=false` (Sync). `prune=true, force=true` is the Hard Sync invocation from the UI.
- `GET /api/v1/argocd/applications/:name/resource-tree` → for the resource-tree panel (Pods, ReplicaSets, Services).

**Auth flow:**
1. Validate the user's IDP session (existing `sessionPlugin`).
2. Read the Dex id_token stored in the session (added in P2.0 — see §4.5.1).
3. Forward as `Authorization: Bearer <id_token>` to `argocd-server.argocd.svc:80`.
4. Argo CD's OIDC verifier accepts it (audience claim contains `argocd` via `trustedPeers`).
5. Pass through the response.

**Error mapping:**
- Argo CD 401 → IDP 401 (user re-auth).
- Argo CD 403 → IDP 403 with `code: "argocd_forbidden"`.
- Argo CD 5xx / network error → IDP 503 with `code: "deploy_plane_unavailable"`.
- Use the existing `mapConnectError` pattern for shape consistency.

#### 4.5.1 Session needs to carry the id_token

Today the session-store plugin stashes only the user_id and a few claims. P2.0 extends it to also persist the raw id_token (encrypted at rest in Dragonfly). On expiry, the gateway short-circuits requests to `/api/v1/argocd/*` with `401 reauth_required` so the UI prompts a reload through `/auth/login`.

**This is the only persistence shape change in P2.0.** No new database, no new tables.

### 4.6 gateway-svc — argocd-notifications webhook

**File:** `apps/gateway-svc/src/http/webhooks/argocd.ts` (new)

**Route:** `POST /api/v1/webhooks/argocd`

**Behaviour:**
1. HMAC verify `X-Argocd-Signature` against the body using the shared secret loaded from `gateway-svc-argocd-webhook` Secret.
2. Parse the JSON body (Zod schema mirroring the §4.3 template).
3. Map argocd-notifications trigger to a NATS subject:
   - `on-deployed` → `idp.deploy.application.synced`
   - `on-health-degraded` → `idp.deploy.application.degraded`
   - `on-sync-failed` → `idp.deploy.application.failed`
   - `on-sync-running` → `idp.deploy.application.running`
4. Publish a CloudEvent (existing envelope shape) to NATS.
5. Respond 204.

**Idempotency:** notification-svc dedupes by `(app, revision, status)` — a duplicate webhook produces a duplicate event with identical key, which gets filtered downstream. No state in the gateway.

### 4.7 packages/events — new subjects + schema

**File:** `packages/events/src/subjects.ts` (changed)

```ts
export const subjects = {
  // ...existing...
  deploy: {
    application: {
      synced:    "idp.deploy.application.synced",
      degraded:  "idp.deploy.application.degraded",
      failed:    "idp.deploy.application.failed",
```

Zod schema (envelope `data` field):
```ts
export const deployApplicationEvent = z.object({
  app: z.string(),                 // service slug (== Argo CD application name)
  revision: z.string(),            // git SHA, short
  syncStatus: z.enum(["Synced", "OutOfSync", "Unknown"]),
  healthStatus: z.enum(["Healthy", "Progressing", "Degraded", "Suspended", "Missing"]),
  operationPhase: z.enum(["Running", "Succeeded", "Failed", "Error", ""]).optional(),
  at: z.string().datetime(),
});
```

## 5. Data flows

### 5.1 Deploy notification path

```
git push to master  ─►  Argo CD reconciler (3-min poll OR push webhook from GitHub if configured)
                        ▼
                        ApplicationSet → Application reconciles its source vs. cluster state
                        ▼
                        Application transitions: OutOfSync → Progressing → Synced+Healthy
                        ▼
                        argocd-notifications fires `on-deployed` trigger
                        ▼
                        POST /api/v1/webhooks/argocd  (HMAC signed)
                        ▼
                        gateway:
                          - HMAC verify
                          - Parse + Zod-validate body
                          - Publish CloudEvent on NATS subject `idp.deploy.application.synced`
                        ▼
                        notification-svc:
                          - Wildcard `idp.>` subscription receives event
                          - Apply per-user authz (every authenticated user gets all deploy events in P2.0)
                          - Push WS frame to all connected sessions of authorized users
                        ▼
                        browser:
                          - EventStreamProvider receives frame
                          - Sonner toast: "catalog-svc: Healthy at abc1234 ✓"
                          - If user is currently on /services/catalog-svc/deployments, refetch app status
```

### 5.2 Sync action path

```
user clicks "Sync" on /services/catalog-svc/deployments
   ▼
   web: POST /api/v1/argocd/applications/catalog-svc/sync
        body: {} (defaults: prune=false, force=false)
        cookie: lw-idp-session
   ▼
   gateway argocd plugin:
     - sessionPlugin validates cookie → user_id, id_token
     - Build Argo CD URL: argocd-server.argocd.svc/api/v1/applications/catalog-svc/sync
     - Forward with Authorization: Bearer <id_token>
   ▼
   Argo CD:
     - Verifies token via Dex (issuer + audience check; `argocd` audience present via trustedPeers)
     - RBAC check: user has `applications, sync, */*` (via role:lw-idp-user)
     - Triggers Application sync; returns 200 with Operation status
   ▼
   gateway: passes Argo CD response back to web (200 + JSON)
   ▼
   web: success toast "Sync requested"; deploy panel will update via WS as state transitions
```

### 5.3 Hard Sync action path

Identical to 5.2 except:
- UI opens `HardSyncDialog`; user confirms.
- Body: `{ prune: true, force: true }`.
- Argo CD issues a sync with prune + force (deletes resources removed from Git, force-applies on conflicts).

## 8. Error handling

| Failure | Behaviour |
|---|---|
| Argo CD pod down | gateway proxy returns `503 deploy_plane_unavailable`; web shows a banner "Deploy plane unavailable — please retry" on deployments tab |
| `argocd-notifications` outage | no real-time toasts; deployments page still pulls live state on render. Graceful degradation. |
| Webhook HMAC mismatch | gateway returns 401, no NATS publish, audit log line. No retries on the gateway side; argocd-notifications has its own retry policy. |
| User id_token expired | gateway returns `401 reauth_required`; web triggers a redirect to `/auth/login` |
| User not in any team that owns the service | P2.0: no team-scoped RBAC, so this case doesn't occur. P1.9/P2.2 will add the gate. |
| Sync triggered while sync in progress | Argo CD returns 200 with the existing operation phase; UI surfaces the message verbatim |
| Hard Sync with no Git source diff | Argo CD treats as a no-op force-apply; UI shows "no changes — force-applied" |

## 12. Open questions / risks

1. **HMAC via argocd-notifications template.** Argo CD's notifications engine supports Go-templated headers but doesn't offer a first-class `hmac` function. Two fallbacks:
   - **A.** Template a static shared bearer token instead of HMAC. Simpler, safe enough for in-cluster traffic (no network egress for the webhook).
   - **B.** Side-car a tiny "signer" container that wraps the webhook payload with a real HMAC before forwarding to the gateway.
   The plan should attempt the bearer-token path first; if audit/security needs HMAC, P1.9 introduces it via Sidecar B.

2. **Session id_token persistence size.** id_tokens are typically 1–2 KB; the existing Dragonfly session is fine. Verify in the plan that we encrypt at rest using the existing session-encryption key.

3. **Argo CD's auto-sync poll interval (default 3 min) is too slow for the demo flow.** Three options:
   - Configure a GitHub webhook → Argo CD's webhook receiver. Adds a webhook setup task.
   - Drop poll interval to 30s. Trades a small load for tighter feedback.
   - Manual Sync click is always available. Sufficient for the demo.
   Default to **30s polling** for the dev cluster; revisit when we move to multi-cluster.

4. **ApplicationSet template DRY.** Listing all 6 service slugs in the ApplicationSet duplicates information already implicit in the `charts/<svc>/` directories. P2.1 may switch the generator to a `git` generator that globs `charts/*/Chart.yaml`. P2.0 ships the literal list to keep moving.

5. **`charts/lw-idp/values.yaml` ownership.** Both Tilt (for inner-loop dev) and Argo CD (for the deployed dev cluster) read this file. As long as Tilt only watches local file changes (it doesn't push to the cluster directly when Argo CD owns the namespace), the two coexist. The plan should verify there's no namespace conflict; if there is, document a workflow ("`tilt down` before letting Argo CD take over the namespace").
