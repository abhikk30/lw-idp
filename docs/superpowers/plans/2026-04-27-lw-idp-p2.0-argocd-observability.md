# lw-idp P2.0 — Argo CD + Deploy Observability Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Tactical task content (failing test → implementation → commit) carried inline per subagent prompt at dispatch time. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Install Argo CD into the dev kind cluster, dogfood lw-idp through it, and turn the portal into the single window into "what's deployed where, at what revision, is it healthy" — with real-time toasts on state changes and Sync / Hard Sync (with confirm dialog) buttons that proxy through the gateway.

**Architecture:**
- **Argo CD as system of record.** No `deployments` table in the IDP. The portal reads via gateway proxy → Argo CD REST API. Argo CD itself watches `master:charts/<svc>/` via an `ApplicationSet` that lists all 6 services.
- **One token model end-to-end.** Dex issues id_tokens with both `lw-idp-gateway` and `argocd` audiences via `trustedPeers`. The session adds an `idToken` field; the gateway forwards it as a bearer to Argo CD on every proxied call. No token-exchange middleware.
- **Real-time push** via `argocd-notifications` controller → gateway webhook → NATS subject `idp.deploy.application.{synced,degraded,failed,running}` → existing notification-svc wildcard subscription → existing WS pipeline → sonner toast in the browser. Zero new persistence.
- **Webhook auth: shared bearer token, not HMAC.** Spec §12.1 — argocd-notifications doesn't natively HMAC bodies; in-cluster traffic doesn't need it. P1.9 can upgrade if needed.
- **Argo CD poll interval 30s** (spec §12.3) — tight enough for the demo, light enough on git provider quota.
- **Bearer routing target:** `argocd-server.argocd.svc:80`. Dev profile only — TLS in P1.9.
- **`charts/lw-idp/values.yaml` ownership:** Tilt watches local files (inner-loop dev); Argo CD owns the deployed cluster. Plan does NOT touch Tilt config — they coexist as long as you `tilt down` before letting Argo CD take over the namespace (documented in the deploy step's commit body).

**Tech stack:** argo/argo-cd Helm chart (current stable, version pinned in helmfile), `argoproj.io/v1alpha1` ApplicationSet + Application, `argocd-notifications-cm` + `argocd-notifications-secret`, Dex 2.41 (existing) with `trustedPeers` cross-audience, Fastify 5 plugin pattern (matches existing http/* modules), Zod schemas (matches `packages/events`), Next.js 15 RSC + TanStack Query (matches P1.7), Radix Dialog (existing in `@lw-idp/ui`), sonner toasts (existing).

## Recon corrections to spec

The implementation engineer should be aware of three details the spec phrased imprecisely:

1. **Session encryption.** Spec §4.5.1 says "encrypted at rest in Dragonfly." This is *Dragonfly's own at-rest encryption*, not an app-layer crypto helper. The session JSON is stored as-is via the existing `SessionStore.set`. **No app-layer crypto code needed in P2.0.** P1.9 can wrap field-level encryption around `idToken` if required.
2. **Argo CD OIDC secret reference.** Spec §4.4 says the secret materializes as `argocd-secret`/`oidc.dex.clientSecret`. The Argo CD Helm chart already manages `argocd-secret` itself — to avoid collision, point `configs.cm.oidc.config.clientSecret` at a *separate* secret via the `$<secret>:<key>` reference syntax. Plan uses `argocd-dex`/`clientSecret` (created by `cluster-bootstrap.sh`).
3. **Event entity/action splitting.** The IDP's WS frame derivation takes the *last two* dot-segments of the NATS subject, so `idp.deploy.application.synced` becomes `entity="application", action="synced"` — not `entity="deploy"`. This affects the invalidation map, the toast text, and the spec-§5.1 sequence diagram caption. Spec is correct in intent; this is just the keying convention.

## Design decisions explicit

1. **Argo CD watches the same monorepo (`abhikk30/lw-idp.git` `master`).** No new GitOps repo. Per-service charts live at `charts/<svc>/`. P2.1 may move to a dedicated `lw-idp-deploys` repo when the bot-PR pattern lands.
2. **`ApplicationSet` lists service slugs literally.** `goTemplate: true` + `list` generator with one element per service (gateway-svc, identity-svc, catalog-svc, cluster-svc, notification-svc, web). DRYing up via a `git` generator that globs `charts/*/Chart.yaml` is deferred to P2.1.
3. **`syncPolicy.automated.prune: false`** by default. Hard Sync is the only path that prunes (via `prune: true, force: true` body to the Sync API). Avoids accidental destructive reconcile.
4. **Argo CD uses cluster's existing Dex** — `dex.enabled: false` in chart values; `configs.cm.oidc.config` points at `http://dex.lw-idp.local`. CoreDNS rewrite already sends that to ingress-nginx.
5. **Single Argo CD RBAC role `lw-idp-user`** mapped to `lw-idp:authenticated` group: `applications get + sync + action/*` on `*/*`. Team-scoped policy is P1.9 / P2.2.
6. **Webhook auth: shared bearer token in `X-Lw-Idp-Webhook-Token` header**, loaded by gateway from a new `gateway-svc-argocd-webhook` Secret. argocd-notifications template injects the token literal into the header. **Not HMAC.** Spec §12.1 fallback A.
7. **Adapter shape change.** P1.7's `DeploymentAdapter` exposes a *historical list* of deploys (`Deployment[]` with `id, environment, status, commitSha, durationSeconds, createdAt`). P2.0's needs are *current state* (one Argo CD Application, not a deploy history). Plan introduces a parallel `ArgoCdAdapter` with `getApplication(slug)`, `listApplications()`, `sync(slug, opts)` — the existing `DeploymentAdapter` mock stays untouched (it'll come back in P2.1 for the build-pipeline timeline). The deployments page rewrites to render the current-state panel from the new adapter.
8. **No `gRPC` surface.** Argo CD speaks REST; the proxy plugin is plain HTTP. We don't add a `deploy.proto`.
9. **Argo CD UI exposed at `argocd.lw-idp.local`** via the chart's built-in ingress block, same pattern as Dex/portal/grafana. CoreDNS rewrite carries it to ingress-nginx in-cluster too, so SSO redirects work for browser AND for gateway-svc proxy lookups.
10. **`@lw-idp/ui` `HardSyncDialog`** wraps the existing Radix `Dialog` primitive (no AlertDialog component exists today). Destructive action button uses `variant="destructive"`.

## Spec deviations — explicit

1. **Webhook signing**: spec §4.6 lists "HMAC verify `X-Argocd-Signature`" as primary; plan ships **shared bearer token** as primary per spec §12.1 fallback A. Reason: argocd-notifications has no first-class HMAC function; the bearer-token literal is materially simpler and in-cluster traffic doesn't need replay protection in P2.0. The webhook plugin is structured so HMAC can be swapped in by adding a single header validator.
2. **Adapter file location**: spec §4.8 lists `apps/web/src/lib/adapters/deployments.ts` as the changed file. Plan instead **creates a new** `apps/web/src/lib/adapters/argocd.ts` with the `ArgoCdAdapter` interface and **leaves the deployments mock alone**. The deployments page swaps which adapter it consumes (from `DeploymentAdapter` → `ArgoCdAdapter`). This preserves the mock's shape for P2.1 reuse.

## Unit test coverage targets

These are the highest-leverage tests; the engineer will add more case-by-case via TDD:

**`apps/gateway-svc/test/http/argocd.test.ts`**:
1. `GET /api/v1/argocd/applications` — session.idToken → upstream `Authorization: Bearer <jwt>`.
2. Same — missing `session.idToken` → 401 `code: "reauth_required"`.
3. `POST /sync` body `{}` → upstream body matches `{ prune: false, dryRun: false, strategy: { hook: { force: false } } }`.
4. `POST /sync` body `{ prune: true, force: true }` → upstream body matches with `prune: true, hook.force: true`.
5. Upstream 401 → IDP 401; 403 → 403; 5xx → 503 `code: "deploy_plane_unavailable"`.
6. Upstream network error → 503.

**`apps/gateway-svc/test/http/webhooks/argocd.test.ts`**:
1. Valid bearer + body for `on-deployed` → NATS publish on `idp.deploy.application.synced` with envelope `data` matching `deployApplicationEventSchema`.
2. Same for `on-health-degraded`, `on-sync-failed`, `on-sync-running`.
3. Missing `X-Lw-Idp-Webhook-Token` → 401, no NATS publish.
4. Wrong token → 401 (constant-time compare).
5. Body missing `revision` → 400 (Zod).
6. Trigger string outside known set → 400.

**`apps/notification-svc/test/authz.test.ts`** (additions):
1. `idp.deploy.application.synced` + authenticated session → allow.
2. Same + undefined session → deny.

**`apps/web/test/components/deployments-panel.test.tsx`**:
1. Renders revision SHA, sync pill, health pill, replicas from initialData.
2. Click Sync → MSW handler called with body `{}` → success toast appears.
3. Click Hard Sync → dialog opens → confirm → MSW handler called with `{ prune: true, force: true }`.
4. Click Hard Sync → cancel → no MSW call.
5. WS frame `idp.deploy.application.synced` for current slug → invalidation → MSW called twice (initial + refetch).
6. 503 from adapter → banner "Deploy plane unavailable" rendered.

## Subagent dispatch format

Each task T receives a prompt (same shape as P1.4–P1.8):

1. **Context block:** 3–5 sentences on what was just done and what T depends on.
2. **Scope block:** files to create/modify from File Structure for T's phase.
3. **TDD loop:** failing test(s) → impl → green → typecheck → conventional commit.
4. **Done criteria:** specific test file(s) green, specific file(s) exist, specific commit SHA reported.

Tasks commit individually. Controller spot-checks against the plan and advances.
