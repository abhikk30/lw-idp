# lw-idp P1.6 — notification-svc + WebSocket fan-out Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Tactical task content (failing test → implementation → commit) carried inline per subagent prompt at dispatch time. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `notification-svc` becomes the real-time edge. A NATS event on `idp.>` arrives, is authorization-filtered per connected user, and is pushed over a WebSocket to the user's browser within 500ms p95. Multi-replica HA; no single point of failure; per-connection backpressure.

**Architecture:**
- **WS fan-out model:** each `notification-svc` replica opens its own non-queue-group JetStream consumer (deliver policy: `new`, ack-none). Every replica receives every `idp.>` message and filters locally against its in-memory `ConnectionRegistry`. NATS is the cross-pod fan-out — no Redis pub/sub layer needed for MVP. Cost: each event decoded N times (N ≤ 3); well under IDP load envelope.
- **Auth model:** `notification-svc` reads the `lw-sid` session cookie **directly** from Dragonfly via the shared `@lw-idp/auth` `SessionStore`. No proxy layer in gateway-svc. Ingress path-routes `portal.lw-idp.local/ws/stream` → `notification-svc` Service; everything else keeps going to gateway-svc.
- **Authorization filter:** pure function `canUserSeeEvent(session, envelope) → boolean` with one rule per known event type, driven off `envelope.actor.{userId|teamId}` + `envelope.data.{owner_team_id|team_id}` + session.teams. Unknown event types: deny by default.
- **Outbound frame:** `{ id, type, entity, action, payload, ts, traceId }` derived from `envelope.type` (split on dots) and `envelope.data`.
- **Per-connection backpressure:** token bucket (100 msgs/sec refill, 50 burst). If drained, drop with counter `lwidp_notification_shed_total`. Prevents a slow client from blocking the consumer loop.
- **Graceful shutdown:** `onShutdown` closes all WS with code 1012 "Service Restart", drains NATS consumer, closes SessionStore — then Fastify closes.
- **SessionRecord.teams trust model:** `session.teams` is **snapshotted at login** and refreshed on next login (session TTL = 8h). Documented explicitly; retires P1.5 IMP-4 without code change. `/api/v1/me` stays live-fetching for accuracy, WS filter uses the snapshot.

**Tech stack:** Fastify 5 + `@fastify/websocket` 11, `nats` client 2.28 (durable consumer), `@lw-idp/auth` SessionStore (reused from P1.5), `@lw-idp/events` envelope + subjects, `ws` 8 for e2e client.

**Spec refs:** `docs/superpowers/specs/2026-04-20-lw-idp-foundation-design.md` §4, §6.6, §7.4, §7.6, §8, §10.2, §11.2.
**Predecessor:** P1.5 (tag `p1.5-gateway-svc`).

## Design decisions explicit

1. **No Redis pub/sub layer in P1.6.** Each pod gets every event from NATS and filters locally. This is cheaper, simpler, and covers a 2–3 pod footprint. If we reach >20 pods we revisit. ADR-011 captures this.
2. **Ingress path-routing over WS proxy.** Add an Ingress rule that routes `/ws/stream` to notification-svc directly. Both services share Dragonfly for session reads. Avoids a hop and the WS-upgrade-through-Fastify-proxy complexity.
3. **No durable consumer state at WS level.** Clients use `last-event-id` reconnect semantics (spec §8.1) which we will implement in P1.7 web portal reconnect logic — not in P1.6. P1.6 ships at-most-once WS delivery for connected clients. Missed messages while disconnected are on the client to fetch via REST (catalog/cluster REST already returns current state).
4. **Consumer deliver policy: `new`.** New clients only see events published after they connect. Avoids replay storms.
5. **Postgres *scaffold only* for notification-svc in P1.6.** Spec §6.6 says "no Postgres" for the fan-out path (honored — the WS path never reads Postgres in the hot loop). But we DO scaffold a `notifications` table + `outbox` table + migration + migrate-cli for **uniformity with identity/catalog/cluster and for P1.7's `ListRecent`/`MarkRead` work** — empty tables, no writes in P1.6. The proto file defines `ListRecent` + `MarkRead` (for codegen stability); gRPC handlers return `Unimplemented` in P1.6.
6. **WS-only, no SSE fallback in P1.6.** SSE was spec-mentioned but WS covers all P1 target browsers (evergreen Chrome/Safari/FF). Deferred.

## Deviations from spec (§6.6) — explicit

1. **Spec says "Dragonfly sticky-routing map (which replica holds a given user's WS)".** We reject this for P1.6 in favor of "every pod receives every NATS event and filters locally" (ADR-011). The sticky-routing map is only needed if you want point-to-point publish from one pod to a specific other pod; we don't, because NATS broadcasts. Revisit past ~20 pods.
2. **Spec says "WS client presents session cookie + short-lived WS token; notification-svc validates with identity-svc".** We simplify to **cookie-only** — notification-svc reads the session directly from Dragonfly via the shared `@lw-idp/auth` `SessionStore`. Same security properties (an attacker without the session cookie cannot connect); one fewer round-trip (no identity-svc hop per connect); one fewer secret to rotate. The short-lived WS token adds defense in depth only if the cookie can leak to a context that can't keep a cookie — not our threat model for P1.
3. **User's original ask said `gateway-svc` exposes `GET /api/v1/ws` and proxies the upgrade to notification-svc.** We chose **ingress path-routing** (`portal.lw-idp.local/ws/stream` → notification-svc Service directly). Trust model is identical (both services read sessions from the same Dragonfly; same top-level host → no CORS/same-origin shift). No proxy code; one fewer hop. If you want to keep the proxy pattern (e.g., because future middleware like audit logging on WS traffic needs gateway-svc as the choke point), flag during review and I'll swap in a `@fastify/http-proxy` upgrade handler — ~1 task swap, no downstream changes needed.

## Unit test coverage targets (authz.test.ts)

Must include at least these cases (to make the filter bulletproof — authorization bugs are the riskiest failure mode):

1. `idp.identity.user.created`, actor.userId = self.userId → allow.
2. Same, actor.userId ≠ self.userId, session in `platform-admins` → allow.
3. Same, neither → deny.
4. `idp.identity.team.member.added`, envelope.data.user_id = self.userId → allow.
5. Same, envelope.data.team_id in session.teams → allow (team member sees the add).
6. Same, neither → deny.
7. `idp.catalog.service.created`, envelope.data.owner_team_id in session.teams → allow.
8. Same, owner_team_id ∉ session.teams, session is platform-admin → allow (admins see all catalog events).
9. Same, neither → deny.
10. `idp.cluster.cluster.registered`, session is platform-admin → allow.
11. Same, session is NOT platform-admin → deny (spec §7.6 — cluster CRUD is platform-admins only; read is authenticated but register-events target admins only).
12. Unknown `envelope.type = "idp.some.new.thing"` → deny (default-deny policy).
13. Envelope with missing `data` → deny (malformed).

## Deliverables

1. A connected WS client receives a matching event within 500ms p95 over localhost (integration test asserts a hard 2s ceiling; we record p95 in log for future SLO alert).
2. An unauthorized user does not receive team-A-owned service.created. (integration test asserts it does NOT arrive within 3s.)
3. 2 pods running notification-svc, client on one pod, event on the other — client receives (proves fan-out isolation correctness).
4. `wscat` live against kind cluster shows frames arriving after a `curl -X POST … /api/v1/services`.
5. Graceful rolling-update of notification-svc disconnects clients with code 1012; client reconnect script recovers.
6. Tag `p1.6-notification-svc`; CI green including the new e2e describe.
7. ADR-011 committed.
8. Gateway-svc on `createRedis()` factory — single Redis client across sessionStore + rateLimit + idempotency + stateStore (retires IMP-1).
