# lw-idp P1.8 — Observability as Code + Storybook + Carry-over Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Tactical task content (failing test → implementation → commit) carried inline per subagent prompt at dispatch time. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn the operating IDP into one we can actually run on-call: per-service Grafana dashboards, SLO burn-rate alerts, Alertmanager routing, and recording rules — all checked into `infra/observability/`. Plus the design-system surface gets Storybook with play tests, and five small carry-over follow-ups from P1.6/P1.7 ship alongside.

**Architecture:**
- **Dashboards as ConfigMaps.** `kube-prometheus-stack`'s Grafana sidecar auto-discovers ConfigMaps labeled `grafana_dashboard: "1"`. Six dashboard JSONs (one per service + platform-overview) ship as a tiny `charts/lw-idp-observability` Helm chart that templates the ConfigMaps + PrometheusRule CRs together. Listed in `infra/helmfile/observability.yaml` so it deploys with the rest of observability.
- **PrometheusRule CRs** for recording rules + alert rules. Recording rules pre-aggregate the per-service `lwidp:requests:rate5m` / `errors:rate5m` / `duration:p95_5m` series so dashboards and burn-rate alerts both query the same cheap series. SLO burn-rate uses the multi-window multi-burn-rate (MWMBR) pattern — fast (5m+1h) and slow (30m+6h) burn windows per spec §11.2 SLO.
- **Per-service ServiceMonitors.** `lw-idp-service` library chart gets a `_servicemonitor.tpl` template (gated `monitoring.enabled`); each service's `values.yaml` opts in. Replaces the nominal "kube-prometheus-stack picks up `/metrics` automatically" fiction with explicit, declarative scraping.
- **Telemetry contract additions.** notification-svc adds `lwidp_notification_fanout_seconds` (histogram) and `lwidp_notification_connections` (gauge); gateway-svc adds `lwidp_gateway_idempotency_replays_total` and `lwidp_gateway_ratelimit_shed_total`. Existing `fastify-metrics`-emitted route/HTTP metrics are kept as-is (renaming would invalidate the Prometheus retention).
- **Storybook 8** in `apps/web/.storybook/` with Tailwind v4 + MSW addon + `@storybook/test-runner` for CI Playwright regression. Six stories with play tests; uses fixtures from `@lw-idp/testing/msw`.
- **Carry-over follow-ups** ride alongside in Phase A: `console.*→logger`, fan-out info log, gateway-side `isSafeRedirect`, theme FOUC, EventStreamProvider 4xxx-class WS-close handling.

## Design decisions explicit

1. **Dashboards live in a tiny new chart, not `infra/observability/` raw YAML.** ConfigMap dashboard discovery requires a label, and Helm makes per-environment differences (e.g., dev vs HA) trivial later. Path: `charts/lw-idp-observability/` with `templates/dashboards/*.yaml` (each templating a JSON file from `charts/lw-idp-observability/dashboards/*.json`). The chart also packages the PrometheusRule CRs.
2. **Recording rules are the source of truth for dashboard queries.** Dashboards query `lwidp:requests:rate5m{service="gateway-svc"}` etc., not raw `http_request_duration_seconds_count`. Two reasons: (1) Prometheus retention compresses recorded series further, (2) burn-rate alerts use the same series and we don't want dashboard/alert drift.
3. **Burn-rate windows: 5m+1h fast (page) and 30m+6h slow (warn).** Standard Google SRE multi-burn-rate. Page = 14.4× burn over 1h means we'd exhaust 30d budget in 2d → page. Warn = 6× over 6h means we'd exhaust budget in 5d → ticket.
4. **Alertmanager** ships a `webhook` receiver pointing at `http://alertmanager-debug-webhook.observability.svc.cluster.local/` (a stub Service we add — just routes to a logging container). Slack/PagerDuty integration deferred to a future ops-readiness pass; this gives us a verifiable end-to-end alert path without external dependencies.
5. **ServiceMonitor opts in per service.** Library template is gated on `.Values.monitoring.enabled`. Each app's `values.yaml` sets `monitoring.enabled: true` + `monitoring.path: /metrics` + `monitoring.port: targetPort`. Avoids accidentally scraping web's Next.js endpoint (no Prometheus metrics there yet).
6. **Storybook ships ONLY for `apps/web`.** No backend services have UI to story. The `@lw-idp/ui` package's primitives are stored from `apps/web/.storybook` so they ship with the web build (consistent with how the web app already imports them).
7. **MSW story decorator** uses the existing `@lw-idp/testing/msw` handlers package — same fixtures unit tests use. No fixture duplication.
8. **`@storybook/test-runner` integration** runs in CI as a dedicated job (`pnpm test-storybook`) — it spins up Storybook on a port and Playwright-runs each story's `play()` function. Adds ~30s to CI but covers the visual + interactive surface that vitest can't.
9. **Grafana ingress** lands at `grafana.lw-idp.local` (already-existing pattern from `portal.lw-idp.local`). No auth — dev only. P1.9 wires Dex SSO. `mkcert` already covers `*.lw-idp.local`.

## Subagent dispatch format

Each task T receives a prompt (same as P1.4–P1.7):

1. **Context block:** 3-5 sentences on what was just done and what T depends on.
2. **Scope block:** files to create/modify from File Structure for T's phase.
3. **TDD loop:** failing test(s) → impl → green → typecheck → conventional commit.
4. **Done criteria:** specific test file(s) green, specific file(s) exist, specific commit SHA reported.

Tasks commit individually. Controller spot-checks against the plan and advances.
