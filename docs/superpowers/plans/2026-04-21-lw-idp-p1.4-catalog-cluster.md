# lw-idp P1.4 — catalog-svc + cluster-svc end-to-end

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Tactical task content carried inline in each subagent prompt.

**Goal:** Both `catalog-svc` and `cluster-svc` become real domain services with Postgres-backed CRUD, ConnectRPC surface, transactional-outbox event publishing. Demo: `grpcurl`-able `CreateService`, `ListServices`, `SearchServices` on catalog; `RegisterCluster`, `ListClusters` on cluster; every mutation produces an `idp.{catalog,cluster}.*` event observable on NATS.

**Architecture:**
- Same shape as identity-svc (service-kit `register` callback, ConnectRPC over Fastify, Drizzle + Postgres per-service database, transactional outbox to NATS JetStream).
- Contracts extended: `lwidp.catalog.v1.CatalogService`, `lwidp.cluster.v1.ClusterService`.
- `catalog-svc` domain: `services`, `service_tags` (GIN full-text), `service_dependencies`, `service_metadata` (JSONB). `SearchServices` uses Postgres FTS on name+description+tags.
- `cluster-svc` P1 scope: registry only (CRUD on `clusters` + `cluster_tags`). Kubeconfig handling + live health defer to a later plan.

**Spec refs:** `docs/superpowers/specs/2026-04-20-lw-idp-foundation-design.md` §6.4, §6.5, §7.4.
**Predecessor:** P1.3 (identity-svc, tag `p1.3-identity-svc`).

## P1.3 review items to retire here

- **I8** — bootstrap secret idempotence: add `kubectl create secret` blocks to `cluster-bootstrap.sh` for all three services (identity-svc-pg, identity-svc-dex, catalog-svc-pg, cluster-svc-pg). Eliminates the "manual at-deploy-time step" drift.
- **I9** — Dockerfile fragility: factor the identity-svc Dockerfile into a template that catalog-svc + cluster-svc use verbatim (copy-file pattern; no dynamic generation needed). Every backend service from here forward uses the same file.
- **I1** — outbox double-publish: switch `publishOutbox` to `SELECT … FOR UPDATE SKIP LOCKED` so multiple replicas don't re-publish the same rows. With two new services joining the outbox pipeline, this can't wait.

## File structure

**Contracts (extend):**
- `packages/contracts/proto/lwidp/catalog/v1/catalog.proto` — `CatalogService` with 8 RPCs; messages: `Service`, `ServiceDependency`, `ServiceMetadata`, plus request/response pairs.
- `packages/contracts/proto/lwidp/cluster/v1/cluster.proto` — `ClusterService` with 5 RPCs; messages: `Cluster` + request/response pairs.
- `packages/contracts/src/index.ts` — re-export new services alongside identity.
- `packages/contracts/dist/proto/lwidp/{catalog,cluster}/v1/*_pb.ts` — generated + committed.

**catalog-svc (`apps/catalog-svc/`):**
- `src/config.ts` — Zod env loader (retires P1.3 review M2).
- `src/db/schema/{services,service_tags,service_dependencies,service_metadata,outbox,index}.ts`.
- `src/db/migrations/` — drizzle-kit generated.
- `src/services/services.ts` — `createService`, `updateService`, `deleteService`, `getService`, `listServices`, `searchServices`, `addDependency`, `removeDependency`.
- `src/grpc/catalog.ts` — `makeCatalogServiceImpl({ db })`.
- `src/grpc/plugin.ts` — mounts ConnectRPC router.
- `src/index.ts` — wires startServer, migrations, outbox publisher.
- `src/db/migrate-cli.ts`, `drizzle.config.ts`, `Dockerfile` (template copy).

**cluster-svc (`apps/cluster-svc/`):**
- Same shape as catalog-svc with 5-RPC surface on `clusters` + `cluster_tags`.

**Shared Dockerfile template:** `apps/_shared/Dockerfile.backend` — single source-of-truth; each service's `Dockerfile` is a symlink or a verbatim copy. We use copies (not symlinks) because Docker Desktop + some Git hosts struggle with them. A `scripts/sync-dockerfiles.sh` enforces byte-identity in CI.

**Helm charts:** mirror identity-svc's `values.yaml` extraEnv pattern. Reuse `charts/lw-idp-service` library chart.

**Infra:** extend `infra/cnpg/init-databases.sh` from `DATABASES=(identity)` → `DATABASES=(identity catalog cluster)`. Extend `scripts/cluster-bootstrap.sh` with the three per-service Secret blocks (I8 fix).

**Tests:**
- Per-service: unit (pure fns) + integration (testcontainers Postgres) + gRPC round-trip (connect-fastify + connect-node client).
- Cross-service e2e stays in `test/e2e/` — add assertions that `CreateService` produces `idp.catalog.service.created` on NATS; `RegisterCluster` produces `idp.cluster.cluster.registered`.

## Deliverables

1. `grpcurl`-equivalent ConnectRPC client can CRUD services and clusters against the live kind cluster.
2. `nats sub 'idp.catalog.>'` shows `service.created`/`updated`/`deleted` within ~100ms of the mutation.
3. `nats sub 'idp.cluster.>'` shows `cluster.registered`/`updated`/`deregistered`.
4. `SearchServices` returns results by name/description/tag prefix + full-text match.
5. CI green on PR.
6. Tag `p1.4-catalog-cluster`.
