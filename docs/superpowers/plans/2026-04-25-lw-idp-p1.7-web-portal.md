# lw-idp P1.7 — web portal UI Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Tactical task content (failing test → implementation → commit) carried inline per subagent prompt at dispatch time. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up the lw-idp web portal as a real Next.js 15 App Router experience over the gateway-svc REST surface and notification-svc WS stream — every route from spec §10.1 rendering, with mocked deploy/pipeline integrations. End-to-end demo: log in via Dex, browse catalog, create a service, see the WS toast fire, see the row appear in the list.

**Architecture:**
- **RSC-first** per spec §10.3. Pages (catalog, clusters, teams, dashboard) are Server Components fetching via server-side `openapi-fetch` against gateway-svc internal DNS (`http://gateway-svc.lw-idp.svc.cluster.local`). Client Components (forms, tables with selection state, command palette, EventStreamProvider) suffix file `*.client.tsx` per convention.
- **Mutations** use TanStack Query mutations + `openapi-fetch` from the browser, hitting `/api/v1/*` through ingress (same origin → cookie auto-attached). Server Actions reserved for one-shot UI flows that don't need optimistic invalidation. (Decision: choose Server Actions for `delete-service` and `delete-cluster` since they're rare, simple, and benefit from server-side error mapping.)
- **WebSocket consumer** (`EventStreamProvider`) opens to same-origin `/ws/stream` (path-routed by ingress to notification-svc, P1.6). Frames carry `{ type, entity, action, payload, ts }`. Provider derives **TanStack Query invalidations** from the `(entity, action)` pair via a static map (e.g. `("service", "created"|"updated"|"deleted") → ["services"]`). One toast per frame using `sonner`.
- **Auth model:** browser hits `portal.lw-idp.local/auth/login` → ingress routes `/auth/*` to gateway-svc (already-shipped P1.5 endpoints) → Dex flow → cookie set on `portal.lw-idp.local`. Web's `middleware.ts` checks the cookie and redirects unauth → `/auth/login`. The browser never sees a JWT; the cookie is HttpOnly. Spec §7.1 + §11.3.
- **Adapter pattern (spec §10.4)** for integration data not yet real. `packages/contracts/src/adapters/{deployments,pipelines}.ts` define the interfaces; `apps/web/src/lib/adapters/` chooses mock impl when `NEXT_PUBLIC_INTEG_DEPLOYMENTS=mock` (default). MSW handlers + fixtures live in `packages/testing/fixtures/` so unit tests, Playwright, and Storybook share one truth.

**Tech stack:** Next.js 15.0.x + React 19 + Tailwind CSS v4 (CSS-first, `@import "tailwindcss"`) + shadcn/ui (manually-vendored components in `packages/ui` — we own them) + TanStack Query v5 + TanStack Table v8 + Zustand 5 + react-hook-form 7 + @hookform/resolvers/zod + sonner 1 + lucide-react 0.4x + cmdk 1 + next-intl 3 (en-only) + openapi-fetch 0.13 + MSW 2 + Vitest + Testing Library + Playwright + axe-playwright.

## Design decisions explicit

1. **`@lw-idp/ui` package owns the design system.** shadcn-wrapped Radix components live there (button, card, dialog, dropdown-menu, input, label, table, toast, tooltip, sheet, command, badge, separator, skeleton). Tailwind v4 preset (CSS-only theme tokens) lives there too. Apps consume via workspace import.
2. **Tailwind v4 CSS-first config (no `tailwind.config.ts`).** Theme tokens declared as CSS custom properties in `packages/ui/src/styles/globals.css` (light + dark via `[data-theme=dark]`). Each app's root layout includes the package's globals.
3. **openapi-fetch with two creators** — one for RSC (server-side, talks to internal DNS, no credentials needed inside cluster) and one for client (browser, same-origin to ingress, credentials="same-origin"). Lives in `apps/web/src/lib/api/{server,client}.ts`.
4. **TanStack Query v5 cache keys are tuples**: `["services"]`, `["services", id]`, `["clusters"]`, `["clusters", id]`, `["teams"]`, `["teams", slug]`, `["me"]`. The WS-event-to-invalidation map keys off these.
5. **Form validation: Zod schemas live in `packages/contracts/src/forms/*.ts`** so they're shared between web (react-hook-form resolver) and gateway-svc (Fastify body validation). For P1.7 we only define `serviceCreateSchema` + `clusterCreateSchema`; richer schemas land later. (Light deviation: gateway-svc currently uses ad-hoc Zod inside its handlers. We don't refactor gateway-svc here — only add the shared schema and use it from web. Gateway-svc adopts in P2.)
6. **Theme + sidebar collapsed state** in Zustand (persisted to localStorage). User profile + teams hydrated server-side from `/api/v1/me` into a non-persisted Zustand slice.
7. **Command palette (`⌘K`)** is shipped with day-one navigation actions only: jump-to-services, jump-to-clusters, jump-to-teams, toggle-theme. No fuzzy entity search yet (that needs a search-svc — out of scope).
8. **MSW operates in browser dev (process.env.NODE_ENV === "development")** AND in Playwright Node mode. Both share fixtures from `packages/testing/fixtures/`. **Production builds do not bundle MSW.**
9. **Settings/api-tokens is a stub.** Renders an empty-state CTA "Generate token (coming P3)" — no backend exists yet.

## Subagent dispatch format

Each task T receives a prompt constructed like this (same as P1.4/P1.5/P1.6):

1. **Context block:** 3-5 sentences of what was just completed and what T depends on.
2. **Scope block:** files to create/modify from the File Structure section for T's phase.
3. **TDD loop:** failing test(s) → implementation → tests green → typecheck → conventional commit.
4. **Done criteria:** specific test file(s) green, specific file(s) exist, specific commit SHA reported.

Each task commits. Between tasks the controller spot-checks against the plan and advances.
